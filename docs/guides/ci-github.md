# Run SigComply in GitHub Actions

How to scaffold and wire the SigComply compliance workflows into GitHub Actions, using GitHub OIDC for both AWS access and SigComply Cloud submission — with no SigComply secret.

> Docs hub: [../README.md](../README.md)

## Prerequisites

- Your code hosted on GitHub, with a working `.sigcomply.yaml` in the repo (see [configure sources](configure-sources.md)).
- The CLI installed locally to run the scaffolder ([install guide](install.md)).
- An AWS IAM role that trusts the GitHub OIDC provider and carries read-only permissions (see [AWS OIDC role](#aws-oidc-role) below). Only needed if you collect from AWS.

## 1. Scaffold the workflows

From your repo root:

```bash
sigcomply init-ci --ci github
```

This writes one workflow per cadence under `.github/workflows/` — exactly these six files:

```
.github/workflows/compliance-daily.yml
.github/workflows/compliance-weekly.yml
.github/workflows/compliance-monthly.yml
.github/workflows/compliance-quarterly.yml
.github/workflows/compliance-annual.yml
.github/workflows/compliance-on-push.yml
```

Exit code `0` means the files were written. `init-ci` refuses to overwrite existing files without `--force`. It is SOC 2-only in this release; other frameworks exit `3`.

## 2. Understand the daily workflow

`compliance-daily.yml` is the representative cadence workflow. It runs on a cron schedule (and on-demand via `workflow_dispatch`):

```yaml
# Generated by `sigcomply init-ci`. Customize freely.
# Daily cadence: most automated checks (MFA enforced, encryption at rest,
# log forwarding, etc.) live here.
name: Compliance — Daily

on:
  schedule:
    - cron: '0 2 * * *'    # 02:00 UTC every day
  workflow_dispatch:

permissions:
  id-token: write
  contents: read

env:
  SIGCOMPLY_VERSION: latest
  AWS_REGION: us-east-1
  AWS_ROLE_ARN: arn:aws:iam::000000000000:role/REPLACE_ME

jobs:
  check:
    runs-on: ubuntu-latest
    timeout-minutes: 30
    steps:
      - uses: actions/checkout@v4
      - name: Install SigComply CLI
        run: |
          set -eu
          version="${SIGCOMPLY_VERSION:-latest}"
          if [ "$version" = "latest" ]; then
            tag=$(curl -fsSL https://api.github.com/repos/SigComply/sigcomply-cli/releases/latest | jq -r .tag_name)
          else
            tag="$version"
          fi
          ver="${tag#v}"
          curl -fsSL "https://github.com/SigComply/sigcomply-cli/releases/download/${tag}/sigcomply_${ver}_linux_amd64.tar.gz" -o /tmp/sigcomply.tar.gz
          tar -xzf /tmp/sigcomply.tar.gz -C /usr/local/bin sigcomply
          chmod +x /usr/local/bin/sigcomply
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ env.AWS_ROLE_ARN }}
          aws-region: ${{ env.AWS_REGION }}
          audience: https://api.sigcomply.com
      - name: Run SigComply
        run: sigcomply check --cadence daily
```

The other cadence workflows are identical except for their `cron` schedule and the `--cadence <name>` argument.

### The `permissions` block does double duty

```yaml
permissions:
  id-token: write
  contents: read
```

`id-token: write` lets the Actions runner mint an OIDC JWT (audience `https://api.sigcomply.com`). That single token is used for **both**:

- **AWS role assumption** — `aws-actions/configure-aws-credentials@v4` exchanges it for temporary AWS credentials via the role in `AWS_ROLE_ARN`.
- **SigComply Cloud submission** — the CLI presents the same token to identify your repo to the dashboard.

No SigComply API key or secret is required. `contents: read` lets `actions/checkout` read your repo.

### Cloud submission auto-enables in CI

When `sigcomply check` runs in CI with an OIDC token available and you have not passed `--no-cloud`, cloud submission auto-enables and posts aggregated **counts only** to `POST /api/v1/runs`. The first submission from a newly connected project starts a 2-month, no-credit-card Pro trial. See [cloud dashboard](cloud-dashboard.md) to connect your repo first.

## 3. Set the environment values

Edit the `env:` block in each workflow (or centralize via repository variables):

| Variable | What to set |
|---|---|
| `SIGCOMPLY_VERSION` | `latest`, or pin to a tagged release such as `v1.2.3` for reproducible runs. Pinning is recommended once you stabilize. |
| `AWS_REGION` | The region your AWS sources live in. |
| `AWS_ROLE_ARN` | The ARN of the IAM role with the GitHub OIDC trust and read-only permissions. Replace the `REPLACE_ME` placeholder. |

## 4. The on-push workflow

`compliance-on-push.yml` runs the **subset** of policies whose `on_push` attribute is true — the fast, code-change-sensitive automated checks (manual evidence is excluded). It triggers on pushes to `main` and on pull requests:

```yaml
on:
  push:
    branches: [main]
  pull_request:

permissions:
  id-token: write    # for OIDC role assumption + SigComply Cloud submission
  contents: read
```

Its run step is:

```yaml
      - name: Run SigComply
        run: sigcomply check --on-push
```

Use this for fast feedback on every change; use the cadence workflows for the full scheduled evaluation.

## 5. Commit and confirm

```bash
git add .github/workflows/
git commit -m "ci: add SigComply compliance workflows"
git push
```

The on-push workflow runs immediately; cadence workflows run at their scheduled cron time (or trigger `compliance-daily.yml` manually via the Actions tab → *Run workflow*). A run exits `0` on pass, `1` on violations (when `ci.fail_on_violation: true` in `.sigcomply.yaml`), `2` on execution error, `3` on config error.

## AWS OIDC role

The `aws-actions/configure-aws-credentials` step assumes an IAM role instead of using long-lived keys. You create, once:

1. An IAM OIDC identity provider for `token.actions.githubusercontent.com`.
2. An IAM role whose **trust policy** allows that provider to assume it, scoped to your repository.
3. Read-only permissions on the role (`ReadOnlyAccess` or a scoped read policy).

Put the role's ARN in `AWS_ROLE_ARN`. For the exact trust-policy JSON and setup steps, follow the AWS documentation on [configuring OpenID Connect in Amazon Web Services](https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_providers_create_oidc.html) and the [`aws-actions/configure-aws-credentials`](https://github.com/aws-actions/configure-aws-credentials) README.
