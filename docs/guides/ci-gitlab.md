# Run SigComply in GitLab CI

How to scaffold and wire the SigComply compliance pipeline into GitLab CI, drive cadence jobs with pipeline schedules, and configure OIDC — including the one edit needed for SigComply Cloud submission to work.

> Docs hub: [../README.md](../README.md)

## Prerequisites

- Your code hosted on GitLab, with a working `.sigcomply.yaml` in the repo (see [configure sources](configure-sources.md)).
- The CLI installed locally to run the scaffolder ([install guide](install.md)).
- An AWS IAM role that trusts the GitLab OIDC provider and carries read-only permissions (only if you collect from AWS). Point to AWS docs for the trust policy.

## 1. Scaffold the pipeline

From your repo root:

```bash
sigcomply init-ci --ci gitlab
```

This writes a single `.gitlab-ci.yml` at the repo root with a shared base job plus one job per cadence. Exit code `0` means it was written; `init-ci` refuses to overwrite without `--force`, and is SOC 2-only in this release (other frameworks exit `3`).

## 2. Understand the shipped pipeline

The base job installs the CLI (curl from GitHub Releases) and mints the OIDC token; each cadence job is gated on a scheduled pipeline plus a `CADENCE` variable:

```yaml
# Generated by `sigcomply init-ci`. Customize freely.

variables:
  SIGCOMPLY_VERSION: "latest"   # pin to a tagged release (e.g. v1.2.3) once stabilised
  AWS_REGION: "us-east-1"
  AWS_ROLE_ARN: "arn:aws:iam::000000000000:role/REPLACE_ME"

.sigcomply_base:
  image: ubuntu:24.04
  id_tokens:
    SIGCOMPLY_ID_TOKEN:
      aud: https://api.sigcomply.com
  before_script:
    - apt-get update -qq && apt-get install -y -qq curl jq ca-certificates
    - |
      version="${SIGCOMPLY_VERSION:-latest}"
      if [ "$version" = "latest" ]; then
        tag=$(curl -fsSL https://api.github.com/repos/SigComply/sigcomply-cli/releases/latest | jq -r .tag_name)
      else
        tag="$version"
      fi
      ver="${tag#v}"
      curl -fsSL "https://github.com/SigComply/sigcomply-cli/releases/download/${tag}/sigcomply_${ver}_linux_amd64.tar.gz" -o /tmp/sigcomply.tar.gz
      tar -xzf /tmp/sigcomply.tar.gz -C /usr/local/bin sigcomply
      chmod +x /usr/local/bin/sigcomply

compliance:on_push:
  extends: .sigcomply_base
  stage: test
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
    - if: '$CI_PIPELINE_SOURCE == "push" && $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
  script:
    - sigcomply check --on-push

compliance:daily:
  extends: .sigcomply_base
  stage: test
  rules:
    - if: '$CI_PIPELINE_SOURCE == "schedule" && $CADENCE == "daily"'
  script:
    - sigcomply check --cadence daily

compliance:weekly:
  extends: .sigcomply_base
  stage: test
  rules:
    - if: '$CI_PIPELINE_SOURCE == "schedule" && $CADENCE == "weekly"'
  script:
    - sigcomply check --cadence weekly

compliance:monthly:
  extends: .sigcomply_base
  stage: test
  rules:
    - if: '$CI_PIPELINE_SOURCE == "schedule" && $CADENCE == "monthly"'
  script:
    - sigcomply check --cadence monthly

compliance:quarterly:
  extends: .sigcomply_base
  stage: test
  rules:
    - if: '$CI_PIPELINE_SOURCE == "schedule" && $CADENCE == "quarterly"'
  script:
    - sigcomply check --cadence quarterly

compliance:annual:
  extends: .sigcomply_base
  stage: test
  rules:
    - if: '$CI_PIPELINE_SOURCE == "schedule" && $CADENCE == "annual"'
  script:
    - sigcomply check --cadence annual
```

- `compliance:on_push` runs on every merge request and every push to the default branch, evaluating the `--on-push` policy subset.
- Each `compliance:<cadence>` job runs only in a **scheduled** pipeline whose `CADENCE` variable matches — you create those schedules in step 4.

## 3. The OIDC token block

The `id_tokens:` block mints a JWT scoped to `https://api.sigcomply.com`:

```yaml
  id_tokens:
    SIGCOMPLY_ID_TOKEN:
      aud: https://api.sigcomply.com
```

GitLab exposes each `id_tokens:` entry as an environment variable of the same name. The CLI's Cloud submitter reads the OIDC token from **`SIGCOMPLY_ID_TOKEN`** (falling back to `ID_TOKEN`), so the scaffolded name works out of the box — no edit needed for Cloud submission. As always, submission only happens in CI when `cloud` is enabled and you don't pass `--no-cloud`.

> **Scaffolded with an older CLI?** Earlier GitLab templates named this token `SIGCOMPLY_OIDC_TOKEN`, which the submitter does not read — so Cloud submission was silently skipped (local vault evidence and AWS collection were unaffected). If your `.gitlab-ci.yml` still has `SIGCOMPLY_OIDC_TOKEN`, rename the `id_tokens:` key to `SIGCOMPLY_ID_TOKEN`, or re-scaffold with `sigcomply init-ci --ci gitlab --force`.

## 4. Create one pipeline schedule per cadence

Cadence jobs only run in scheduled pipelines. In your GitLab project:

1. Go to **Settings → CI/CD → Pipeline schedules → New schedule**.
2. Set the cron interval for the cadence (e.g. `0 2 * * *` for daily).
3. Under **Variables**, add `CADENCE` with the matching value: `daily`, `weekly`, `monthly`, `quarterly`, or `annual`.
4. Save. Repeat for each cadence you want to run.

Each scheduled pipeline triggers only the job whose `rules:` match its `CADENCE` value.

## 5. Set the pipeline variables

Edit the `variables:` block (or set them as project CI/CD variables):

| Variable | What to set |
|---|---|
| `SIGCOMPLY_VERSION` | `latest`, or a pinned tag such as `v1.2.3`. |
| `AWS_REGION` | The region your AWS sources live in. |
| `AWS_ROLE_ARN` | The ARN of the IAM role with the GitLab OIDC trust and read-only permissions. Replace `REPLACE_ME`. |

For the AWS OIDC trust policy, follow the AWS documentation on configuring OpenID Connect identity providers; put the resulting role ARN in `AWS_ROLE_ARN`.

## 6. Commit and confirm

```bash
git add .gitlab-ci.yml
git commit -m "ci: add SigComply compliance pipeline"
git push
```

The `compliance:on_push` job runs on the push; cadence jobs run when their schedules fire. Exit codes: `0` pass, `1` violations (with `ci.fail_on_violation: true`), `2` execution error, `3` config error.
